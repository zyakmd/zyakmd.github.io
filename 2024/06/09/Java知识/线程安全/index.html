多线程安全问题主要就是 原子性、可见性、有序性

# java 中

由于并发会迫使原子性丢失，java i++ 编译成汇编有 3 步，1. 内存读 2. 修改 3. 写进内存
加锁，就是转为串行，资源只允许当前线程获取

# 功能角度

共享锁：读锁

排他锁：写锁

# 性能角度

所以要区分锁粒度，也就是安全和性能的平衡（尽可能只锁定影响的部分 / 无锁，乐观锁 CAS）

加锁会导致性能问题：

  1. 竞争状态，上下文切换 mutex 机制，也无法改变
  2. 线程的阻塞和唤醒
  3. 最开始说的并串行问题 起因，无法处理

因此诞生了 偏向锁，轻量级锁，重量级锁。基于锁的本质，就是竞争一个同步状态，用标识（内核中的 mutex 机制，会有内核 / 用户态的切换问题）。

  • 有时候不存在竞争，线程获取时就是偏向锁，加一个偏向而非加锁，下次再来，这个偏向还在，就不用加锁
  • 通过重试，解决进入阻塞，也就是自旋锁 ，即轻量级锁。不用加锁，而是重试等待。轻微竞争
  • 重量级锁，那就是阻塞了
  1. 初始状态是无锁
  2. 当有一个线程加锁，此时偏向锁，就算执行完，对象头里面的偏向线程 id 并不置空
  3. 有另外一个线程尝试进入，发现对象头里面已有 id，升级为轻量级锁，CAS 尝试修改当前线程 ID
  4. 轻量级锁状态下，通过判断对象头中是否有指向锁记录的指针得到该锁对象是否被某线程占有，若占有，自旋 cas 操作修改这个指针，当自旋的线程数过多时，升级为重量级锁
  5. 重量级锁状态下，不再自旋，此时获取不到锁的线程进入内核态。重量级锁状态下，通过 monitor 控制线程

编译器对此优化：

  • 锁消除：发现其实没有安全，消去锁

  • 锁膨胀：发现频繁竞争，升级锁

  • 悲观锁

  • 公平 / 非公平锁

# 特性角度

  • 重入锁：防死锁
  • 分布式锁：redis /zookeeper/ 锁表 实现
