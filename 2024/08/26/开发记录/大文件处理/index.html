# 环境

Spring Boot、mybatis、pagehelper

# 前情提要

工作要到了一个记录行非常大的文件的情况，百万行一个文件就上 G，虽然用线程池 + 异步 + 批量的方式解决了出入库，但使用 pagehelper 的分页效率还是不够高， 跳一页要一两秒，明显受到了 limit offset 原生方法的拖累。故此记录针对这两个问题的优化方案 + 数据结果。

# 出入库

朴实的 线程池 (异步) + 批处理，取得了能有缩到 20% 左右的效果。
又看了说：LOAD DATA INFILE 比批处理还顶。但是 LOAD DATA INFILE 必须要文件数据格式和表结构对应。如果特别需要才使用，还要定制文件格式才行。

# 分页读取

解决思路：最直接就是换数据库，用大数据的 ElasticSearch，或者缓存式的 redis，又或者设计上分库分表。但环境特殊，就针对 Mysql 而言，其实就是慢查询问题，之前有了解过，但没实际上手过这么大量的数据，这次练练。

具体方案：

  1. 用索引范围查询，但一般都不会完全自增了，而且删除时会有留空，pass
  2. 子查询 id 先，再获取全数据。看着绕了一下，但实际全扫描处理的数据少很多，从缓存的角度一下子就少了很多，应该是可以降低磁盘的数据交换开销？和提高 buffpool 异曲同工？ √
    1. 有效
    2. 加大缓存呢？
      1. 也有效，而且曲线平稳。两者其实原理一致就是尽可能降低磁盘开销，我觉得这也是 Mysql 的核心问题，Redis 缓存不用磁盘嘛，就是快。
  3. PageHelper.getLocalPage().setCount(false); 如果不需要总数的话可以不用开这个，因为 pagehelper 每次都要查询一次 count，慢很多。需要前端配合 / 缓存起来。

效果还是比较明显的，百万分页叶只用几百毫秒。
