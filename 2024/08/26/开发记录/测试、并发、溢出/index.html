# 测试

  • QPS（Query Per Second） ：服务器每秒可以执行的查询次数；
  • TPS（Transaction Per Second） ：服务器每秒处理的事务数（这里的一个事务可以理解为客户发出请求到收到服务器的过程）；
  • 并发数 = QPS /TPS * 平均响应时间 (RT)

如，访问一个页面会请求服务器 2 次，一次访问，产生一个 “T”，产生 2 个 “Q”。

使用 jmeter 进行压测 ，使用 NIO 和 线程池提高 TPS。

# 1. TPS（Transactions Per Second）

  • 定义：每秒钟可以处理的事务数量，尤其是在交易系统中（如证券交易、支付网关等）。
  • 重要性：TPS 是金融系统的核心指标之一，因为许多金融应用涉及大量的交易处理，要求系统能够在短时间内处理大量交易，确保高效和准确。

# 2. 响应时间（Latency）

  • 定义：从发送请求到收到响应的时间间隔。
  • 重要性：金融系统中的延迟可能导致严重的后果，如交易时机的错失。因此，低延迟是交易系统、在线支付系统和市场数据处理系统中的关键指标。

# 3. 吞吐量（Throughput）

  • 定义：系统在单位时间内能够处理的请求或交易数量，通常以字节数或数据包数来衡量。
  • 重要性：对于需要处理大量数据的金融系统（如批量结算、清算系统等），高吞吐量可以确保系统能够在规定的时间窗口内处理所有必要的数据。

# 使用逻辑

  • 响应式（非阻塞）：

  • 这个方法返回 Mono<String> ，实际的操作是异步执行的，返回的 Mono 代表将来会有一个 String 结果。

  • 非响应式（阻塞）：同样兼容传统的模式，就是失去了异步效果

  • 这个方法直接返回 String ，方法调用是同步和阻塞的，直到生成 String 结果。

# 原理

调用一个方法后，立即返回一个 Mono 或 Flux 对象，实际的处理可能在后台异步进行。当数据可用时，通过回调机制通知订阅者。

# 总结

核心： 请求超过了处理能力（各个方面都有可能）

  1. 提高服务器
    1. 线程池
    2. NIO
  2. 提高数据库本身负载能力
    1. SQL 优化
    2. 调参，Buff_Pool
  3. 代替数据库承载
    1. 缓存 /redis 等
    2. 要避免击穿，且高并发下第一次可能要加锁（不然就是大量并发漏过去了）
  4. 限制请求数量
    1. kafuka 慢消费
    2. 服务解耦分流
  5. 网络
    1. 流量过大，拥塞
