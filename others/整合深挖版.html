# 技术栈

  • Spring 框架
    • Spring，--MVC，--Boot 的关系和区别
      • Spring
        • 核心容器（Core Container）：包括核心、上下文、Bean 工厂等组件，提供了 IoC（控制反转）和 DI（依赖注入）功能。
        • AOP（面向方面编程）：支持面向方面的编程，允许通过声明切面来实现横切关注点（如日志记录、事务管理等）。
        • 数据访问 / 集成：包括 JDBC、ORM、事务管理等模块，简化了数据访问和集成。
        • Web 模块：包括 Spring MVC，提供了创建基于 Web 应用程序的支持。
        • 其他：如安全（Spring Security）、批处理（Spring Batch）、消息（Spring Messaging）等模块
      • Spring MVC 是 Spring 框架的一部分，主要用于构建传统的 Web 应用程序，它提供了强大的请求处理和视图渲染功能
      • Boot 是基于 Spring 框架的一个项目，旨在简化 Spring 应用程序的创建、配置和部署。主要是自动配置、嵌入式服务器、独立运行的 JAR、
    • 常见注解
      • Configuration：配置类。Bean、ComponentScan、Component、Service、Repository
      • Controller、Autowired、@Qualifier：在自动注入时，指定要注入的 bean 的名称，避免歧义。@Resource：可以指定 bean 名称和类型进行注入。@Value：${}。Transactional
      • MVC：@RequestMapping、@GetMapping、@PostMapping、@PutMapping、@DeleteMapping、@RequestParam、@RequestBody、@PathVariable、
    • 初始化过程
      • 创建 SpringApplication 实例：调用 SpringApplication.run(Class<?> primarySource, String... args) 方法启动 Spring 应用
      • 配置 SpringApplication：设置默认的监听器、初始化器、主类等。
      • 准备环境：准备并配置 Environment ，包括处理配置文件和环境变量。
      • 创建并刷新 ApplicationContext：用 refresh() 方法初始化上下文，加载 bean 定义、注册 bean、启动生命周期处理器等。Tomcat 也在这启动
      • 加载和注册 Beans：将 bean 定义注册到 BeanFactory 中。
      • 完成应用程序启动：在应用上下文刷新之后，但在 run() 方法返回之前，执行所有 CommandLineRunner 和 ApplicationRunner 实现类中的逻辑
    • 事务
      • 如果一个事务 a 里开了线程 b，执行别的，ab 能保证原子和一致性吗？不能
      • 本质是每个线程的 ThreadLocal 有自己的 connection，不同的数据库连接导致
      • 一个是分布式事务（二阶段、三阶段提交）。还有就是编程式事务，在 configuration 注解写个 bean 自定义连接类（datasource 作为入参），使用时通过 Autowired 自动注入这个类，通过这个类 getTransaction 开启事务，try（写执行的逻辑 + 提交），catch（回滚） / Spring 实现了一个 TransactionTemplate
    • 定时器相关（ Timer 单线程的定时器类 [有任务积压的问题]， ScheduledExecutorService 多线程的， Spring Task 可以通过 cron 实现具体时间的，）
      • 数据库定时加载到 redis 应该怎么保证连续性？！
        • 双缓存机制
        • lua 脚本执行，事务
        • 重试，状态记录
        • 幂等性
      • 多个定时任务怎么保证不冲突？（单机就如上面说的）
        • 分布式定时任务（redis）：通过 ZSet 的方式 和 键空间通知的方式
        • 将定时任务存放到 ZSet 集合中，并且将过期时间存储到 ZSet 的 Score 字段中，然后通过一个无线循环来判断当前时间内是否有需要执行的定时任务，如果有则进行执行
        • 给所有的定时任务设置一个过期时间，等到了过期之后，我们通过订阅过期消息就能感知到定时任务需要被执行了，此时我们执行定时任务即可。（只能设定一个类型，不能单个，要做业务逻辑）
        • Redis Pub/Sub
      • 重复执行问题
        • 单点机器
        • 建表，以字段为标识作为分布式锁
        • Redis + 过期机制，实现一段时间内的分布式锁
        • Spring Task 的话有像 quartz 的框架可以使用
  • MySQL 优化
    • 分页查询（范围查询，扩大缓冲区，客户端限制【每页最后的 id 传过来】、分库分表）
      • 对限制列加索引，但本质要回表，当偏移量大的时候依旧不行
      • 时间戳 + 序列号，加上可排序的字段
        • 若有序，正序大，就反过来查
      • 组合主键 / 联合索引（字段不覆盖时就失效）
      • 先查 id，再查具体字段
      • 滚动归档（都需要时，联合查询开销不可避免，但假如能拆开任务，并且允许等待就还好）
      • 分区（将大表的数据分成多个小块（分区），每个分区独立存储，从而优化数据的查询、插入和删除操作），最基础设计，按时间分区
    • UUID、雪花算法的算法原理
      • UUID 的长度为 128 位，V1 由时间戳（60 位）、时钟序列（14 位）和节点（48 位）组成，还有纯随机、带哈希的
      • 生成的 ID 为 64 位整数（Java 中的 long 类型），它的设计目标是生成有序且高效的分布式唯一 ID。时间戳、数据中心标识、机器标识、序列号（同一毫秒内生成的不同 ID 的序列号，12 位，能有 4096 的差异，还不行就等待了）缺点是依赖系统时钟，时间回退可能导致问题
  • Redis 使用
    • 效率问题：底层 C 语言，内存操作， 非阻塞的 IO 复用模型机制 epoll （也是如何实现一对多的原理），单线程避免上下文切换，跳表的优化操作
    • 也用 TCP 连接，以及 pipeline 降低网络波动的影响（一次传多条指令，但出错仍会执行完所有返回各自的结果）
    • RDB、AOF 又或者后台清理的时候其实还是开了子线程
    • 实现消息队列：
      • 一般使用 list 结构作为队列， rpush 生产消息， lpop 消费消息。当 lpop 没有消息的时候，要适当 sleep 一会再重试。
      • 可不可以不用 sleep 呢？list 还有个指令叫 blpop ，在没有消息的时候，它会阻塞住直到消息到来。
      • 能不能生产一次消费多次呢？使用 pub /sub 主题订阅者模式，可以实现 1:N 的消息队列。
      • pub /sub 有什么缺点？在消费者下线的情况下，生产的消息会丢失，得使用专业的消息队列如 rabbitmq 等
    • 集群：Redis Cluster
      • 单地将 16384 个哈希槽均匀分配给集群中的节点
      • 支持单节点单机模式的，也支持一主多从的主从结构，还支持带有哨兵的集群部署模式（单机单节点的时候就类似数据库分库分表？一主多从就类似读写分离，哨兵也是后者）
  • 锁
    • 就看线程安全那块，理解锁的升级和降级
  • 并发
    • 线程池基本没问题了，阻塞队列也了解，想点更业务的场景
    • 大小，几个
    • 分布式事务避免冲突
      • 两阶段提交（2PC）、三阶段提交（3PC）、Saga 模式和基于消息队列的事务
      • 前两者都要选出协调者，同时 2PC 有阻塞和单点故障的问题
  • Zookeeper
    • 序列化的原理
      • JSON：对象被序列化为一个字符串，字符串中包含了对象的属性名和值。是文本格式，相比于二进制格式会占用更多的存储空间和带宽
      • Hessian：二进制 + 标识符
    • 监听的原理
      • 在 Client 有 connect 线程和 Listener 线程，通过 connect 线程将注册的监听事件发送给 Zookeeper。将注册的监听事件添加到 Zookeeper 的注册监听器列表中，Zookeeper 监听到有数据或路径发生变化时，把这条消息发送给 Listener 线程，Listener 线程内部调用 process () 方法
    • 怎么使用的（节点的值 key-value？）
      • 节点本质就是路径，每个路径下有一个节点信息。
      • 使用 Curator 的包实现节点的封装管理，可以指定服务名，ip，端口等信息，创建 serviceDiscovery 指定根节点，后续注册都在这之下 ServiceInstance 使用子节点
      • 使用时，通过 ServiceInstance<T>builder 创建该节点时，会在自动生成一个随机的 UUID，是否服务相同是根据这个 id 来看的，因此按服务名获取列表并没有问题。
    • 通知订阅，和 redis 的有什么差别吗？
      • zookeeper：
        • 客户端可以在读取节点数据或获取节点列表时设置一个 Watcher。
        • 当数据节点发生变化（如节点数据更新、节点删除、子节点变化）时，ZooKeeper 会触发 Watcher 通知客户端
        • 一次性的，触发后，Watcher 就会失效，需要重新设置
      • Redis：Pub/Sub 和 Streams 用于消息队列和推送
  • RocketMQ
    • 是将存储请求，还是指令之类的？ 就好比 Netty，发送的是包装好的，RocketMQListener<T> 指定对应的泛型，注解上标明 topic 和 group
    • 相比 kafaka，rocketMQ 源自 kafaka，但架构简化了（去掉 zookeeper，设置 commitLog），但强化了功能（支持 tag，在 MQ 上就进行消息过滤，延时队列，事务，失败队列）

# 数据库上

关系型，MYSQL，ORECLE。MongoDB 和 ElasticSearch 都是非关系型，文档型（数据存储在集合中，每个集合包含多个文档）的（数据库、搜索引擎，一个管理数据，一个检索数据）es 追求写入吞吐量（有默认延迟）, MongoDB 读写比较均衡

MongoDB：

  • B 树或 B + 树作为索引的数据结构
  • 键是文档中的某个字段值，而值通常是一个指向包含该键的文档的指针（直接定位了文件，而不像 MYSQL 节点存数据）
  • 内存映射：WiredTiger 使用内存映射文件来提高数据访问速度。
  • 压缩：支持数据压缩，减少磁盘空间占用。
  • 文档分配和自由列表：文档的插入和删除会动态管理空间，减少碎片化

redis 怎么分页？

  • Zsort / List

为什么用 zookeeper？

  • 分布式协调和一致性需求高：如果项目需要强一致性和可靠性的分布式协调服务，例如分布式锁、选举等，ZooKeeper 是一个很好的选择。
  • 微服务架构需求：如果项目需要构建复杂的微服务架构，包括服务注册与发现、负载均衡、断路器等功能，则 Spring Cloud 是更适合的选择。
  • Eureka：Spring Cloud Netflix 提供的服务注册与发现组件，是一个基于 REST 的服务，可以作为微服务架构中的注册中心，支持服务注册、发现和负载均衡。
  • Consul：HashiCorp 提供的开源工具，支持服务发现、健康检查和 KV 存储等功能，也可以作为 Spring Cloud 中的注册中心。

为什么用 rocketMQ？

  • 顺序。Kafka 也可以通过分区和分区键来实现部分顺序性，但不如 RocketMQ 那样强调顺序消息的支持。同一个消息队列（Topic 的某个 Queue）的消息按照发送顺序被消费者顺序处理。
  • 事务。发送方需要实现事务监听器（TransactionListener），在发送事务消息时执行本地事务，并根据事务执行结果通知 RocketMQ 是否提交或回滚该消息。RocketMQ 提供了事务状态检查器（Transaction State Checker），定期检查处于未确定状态的事务消息
